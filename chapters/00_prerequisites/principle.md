# 00 前置小课：先搭一段坡，再进 Newton

## 0. 为什么先有这一章

你不需要先学完整套机器人动力学，也不需要先把 CUDA、Warp、Newton 文档全刷完，才有资格开始读后面的内容。

这章只做一件事：先给你搭一段很短的坡，让你能顺着进入 `01` 和 `02`。它不是独立大课，也不是百科索引。

读完这一章后，你只需要达到三个很小的目标：

- 知道“一步仿真”通常在做什么。
- 见到 `articulation`、`M`、`J`、ABA、CRBA 这些词时，不会完全失去方向。
- 见到 `wp.array`、`wp.launch`、`wp.tid()` 这些 Warp 词时，能先把它们放进一个粗略执行模型里。

如果你已经在读一个 Newton 例子，那就把这章当成补洞页：哪里卡，就补哪里，然后继续往前走。

## 1. 先建立“仿真一步”的通用直觉

先不要急着想 Newton，也先不要急着想某个具体 solver。对大多数仿真系统来说，一步更新都可以先拆成四个很通用的问题。

| 这一步要回答什么 | 先用最朴素的话去理解 |
|---|---|
| 静态描述 | 世界里有哪些东西，它们怎么连着，哪些属性通常不会每一帧都重建。 |
| 当前状态 | 这一刻它们在哪、朝哪动、有哪些会随时间变化的量。 |
| 外部输入 | 这一拍你给系统什么命令、力、目标或扰动。 |
| 一步更新 | 按规则把“这一刻”推进到“下一刻”。 |

这四件事连起来，就是最普通的“仿真一步”。你可以把它想成：先知道系统长什么样，再读当前快照，再看这一拍施加了什么，然后算出下一拍。

这个直觉现在比公式更重要。因为你读后面的 Newton 代码时，首先需要分清的是“哪个对象是在描述世界”“哪个对象是在保存当前帧”“哪个对象是在施加输入”“哪个对象是在推进一步”，而不是立刻推导完整动力学。

到了 `02`，这套通用直觉会落到一组更具体的名字上：

- 静态描述，对应 `Model`
- 当前状态，对应 `State`
- 外部输入，对应 `Control`
- 一步更新，对应 `Solver`

先记住这个对应关系就够了。你现在还不需要先吃透 Newton 的全部实现，只需要知道 `02` 其实是在把这四层落到真实代码入口上。

## 2. 刚体与机器人动力学最小词汇

这一组词不要当成散着背的 glossary 项。更有用的读法是：把它们当成一条最小阅读链，也就是第一遍阅读时只需要抓住的先后关系。你不是在收集六个定义，而是在搭一条能从 `00` 走到 `02`，再继续分流到 `05` 和 `07` 的小坡。

这一节也不是让你现在开始推刚体动力学。你暂时不需要推导，不需要背矩阵公式，也不需要先学某个 Featherstone 算法的步骤。现在真正需要的是：每个词大概在回答什么问题，它为什么会和前后词连在一起。

第一遍阅读时，路线只要记成三站：

- 先去 `02_newton_arch`：把这些词挂到 Newton 的 `Model / State / Control / Solver` 和 solver 地图上。
- 再去 `05_rigid_articulation`：真正展开 articulation、generalized coordinates、`M`、ABA、CRBA。
- 最后去 `07_constraints_contacts_math`：真正展开 `J` 和约束 / 接触这条线。

把它们串成最小链时，先后关系可以直接看成四步：

- `articulation`：先分清你描述的是哪一整套关节系统。
- generalized coordinates：再分清你用哪些自由度描述它。
- `M` / `J`：然后认清全局惯性对象和空间映射桥梁。
- `ABA` / `CRBA`：最后把它们当成会使用这些结构的算法名。

### `articulation`（多刚体系统）

`articulation` 先解决的是“我现在面对的到底是什么级别的系统”。如果只是单个刚体，你可以直接谈它的位置、速度和受力；但一旦很多刚体通过关节连起来，问题就变成了“整台机构怎样作为一个系统被描述”。机械臂、双足机器人、车杆系统，读起来通常都属于这个层级。

所以它是后面所有词的容器。广义坐标不是在描述某一块零件，而是在描述这条 `articulation` 的自由度；`M` 和 `J` 也不是零散地属于某个局部，而是在谈这整套关节系统的惯性和映射关系。

你在 `02` 里会先把它当成 Newton 刚体主线里的背景词，第一次真正深入是在 `05_rigid_articulation`。现在先把它稳稳理解成“由关节连接的多刚体系统”就够了，不必提前进入刚体树、空间向量或推导细节。

### generalized coordinates（广义坐标）

广义坐标解决的是“我用什么变量来描述这套系统现在是什么姿态”。如果你硬盯每个刚体在世界里的完整位置和朝向，描述会很重，也不贴合关节系统真正的自由度。对关节系统来说，更自然的语言通常是关节角、关节位移这类最小自由度变量。

一旦你接受系统是用这组自由度来描述的，后面的 `M` 和 `J` 就有了共同语言。`M` 会在这组坐标上谈惯性怎样组织，`J` 会在这组坐标上谈自由度的变化怎样影响末端、接触点或任务空间。

你在 `02` 里会先把它当成理解 `State` 里 articulation 构型与速度表示的背景词，第一次更认真地遇到它是在 `05_rigid_articulation`。现在先知道“它是在描述系统构型和自由度”的语言就够了，不需要先讨论最小坐标的所有细节。

### mass matrix `M`（质量矩阵）

质量矩阵 `M` 解决的是“如果我从广义坐标视角看整个系统，惯性到底怎样被整体组织起来”。它让你不再只用“某个零件有多重”去想，而是开始用“这套自由度整体有怎样的惯性耦合”去想系统反应。

所以 `M` 是接在广义坐标后面的。先有系统自由度，才谈得上这些自由度的惯性耦合怎样形成一个整体对象。后面一些算法会直接用到 `M`，而 CRBA 更是专门和“怎样得到 `M`”这件事连在一起。

你在 `02` 里暂时只要认得 `M` 是刚体主线里的全局惯性对象，第一次真正深入主要在 `05_rigid_articulation`。现在不必进入矩阵装配、推导或实现细节。

### Jacobian `J`（关节雅可比）

`J` 解决的是另一类问题：不是“惯性怎么组织”，而是“关节空间和任务空间、接触空间之间怎样互相对上号”。当你想问“这组广义速度会怎样映到末端或接触空间”，或者反过来“这些空间里的量怎样回到关节自由度”时，就会自然遇到 Jacobian。

这也是为什么 `J` 经常和末端、接触点、约束放在一起讲。和 `M` 相比，`M` 更像在回答整体系惯性，`J` 更像在做空间之间的桥接。它把前面的广义坐标往外连到任务和接触问题上。

你在 `02` 里只需要先把它认成刚体与约束阅读里的桥梁词；真正更认真地见它，通常是在 `07_constraints_contacts_math`。现在先停在“它负责映射关系”就够了，不用展开 Jacobian 公式或完整约束数学。

### ABA

ABA 解决的是“已知这条 `articulation`、当前 `State` 和 `Control` 后，怎样高效算出系统会产生什么加速度”。对现在的阅读来说，先把它看成 Featherstone 路线里的核心算法名，比背全称更重要。

它之所以排在 `M` / `J` 后面，是因为算法不是凭空出现的。只有先有系统结构、自由度和惯性这些背景，你才会开始问“那正向动力学到底怎么高效算”。ABA 就是在回答这类问题。

你在 `02` 里会先把它当成 rigid solver 地图背后的关键词，第一次真正展开是在 `05_rigid_articulation`。现在先知道“ABA 是前向动力学算法名，不是新的物理对象”就够了。

### CRBA

CRBA 解决的是“如果我确实需要整体系的质量矩阵 `M`，怎样高效把它构出来”。所以它和 `M` 的关系最直接。它不是另起炉灶的新物理概念，而是在回答如何得到那个全局惯性对象。

它常和 ABA 一起出现，是因为两者都属于 articulation 主线里的经典算法名，但分工不同：ABA 更偏“算系统响应”，CRBA 更偏“构造 `M`”。把这两件事先分清，后面读算法名字时就不容易混。

你在 `02` 里只需要先认得这个名字，第一次真正深入也主要在 `05_rigid_articulation`。现在先记住“CRBA 是高效构造 `M` 的算法名”，不需要提前进入算法步骤。

把这六个词按这条顺序看，你到 `02` 再见到它们时，就不会觉得它们是零散黑话。下面这张表只是把这条链再压成一页速查。

| 词 | 它主要回答什么问题 | 在这条链里扮演什么角色 | 第一次深入章节 |
|---|---|---|---|
| `articulation` | 我描述的是单个刚体，还是一整套由关节连接的系统？ | 这条链的系统对象，也是后面所有词的容器。 | `05_rigid_articulation` |
| generalized coordinates | 我用什么自由度来描述这套系统的构型？ | `M` 和 `J` 的共同语言。 | `05_rigid_articulation` |
| mass matrix `M` | 这套自由度的惯性怎样整体组织起来？ | 给出整体系惯性的全局对象。 | `05_rigid_articulation` |
| Jacobian `J` | 关节空间怎样连到末端、任务或接触空间？ | 做速度与力映射的桥梁。 | `07_constraints_contacts_math` |
| ABA | 已知结构和输入后，怎样高效做正向动力学？ | articulation 主线里的前向动力学算法名。 | `05_rigid_articulation` |
| CRBA | 如果需要 `M`，怎样高效把它构出来？ | articulation 主线里的 `M` 构造算法名。 | `05_rigid_articulation` |

## 3. GPU / Warp 最小直觉

如果你平时更习惯 CPU 思路，可以先从一个普通 `for` 循环开始想。

在 CPU 上，你可能会先写成这样：

```python
for i in range(n):
    out[i] = f(x[i])
```

Warp 的第一层心智切换是：不是让一个循环把 `n` 个元素一个个做完，而是把“对单个元素做什么”写成一个可批量执行的 kernel，然后把很多元素一起发出去。

```python
@wp.kernel
def compute(x: wp.array(dtype=float), out: wp.array(dtype=float)):
    i = wp.tid()
    out[i] = f(x[i])

wp.launch(compute, dim=n, inputs=[x], outputs=[out])
```

读这段时，先抓住下面几件事：

- `kernel`：不是更神秘的函数，本质上是“单个元素该怎么处理”的批量模板。
- `wp.array`：不是普通 Python list；先把它当成“kernel 能稳定读写的数据缓冲区”。
- `wp.launch`：不是在写逻辑，而是在说“把这个 kernel 按多大规模真正跑起来”。
- `wp.tid()`：就是“这次轮到我处理哪个元素”的索引入口。

`wp.atomic_add` 出现时，通常说明问题从“每个线程各写各的格子”变成了“很多线程可能同时往同一个位置累计结果”。如果没有原子加，你就会碰到写冲突。第一遍只要建立这个直觉，不必现在展开并发细节。

`wp.tile` 可以先浅浅理解成“为了局部复用而把一小块数据放在更适合一起处理的地方”。你现在只要知道它服务的是局部性和吞吐，不需要先理解底层优化策略。

`Graph` 也先浅浅理解就好：如果一串 `launch` 会反复重复执行，把它们和依赖关系打包起来，可以减少重复调度的开销。第一遍不需要先进入 capture、replay 这些更细的机制。

这一节的目标不是把 Warp API 背完，而是让你进 `01` 时不再把每个名字都当成陌生黑箱。

## 4. 现在卡住时，应该先跳去 01 还是 02

你接下来通常只需要做一个分流：你卡在的是 Warp 执行模型，还是卡在 Newton 自己的对象分层。

| 如果你现在更像这样卡住 | 先去哪里 |
|---|---|
| “我知道仿真每步在更新，但我一看到 `kernel`、`wp.launch`、`wp.tid()`、`wp.atomic_add` 就断线。” | 先去 `01_warp_basics` |
| “我大概知道 GPU 是批量执行，但我不知道 Newton 里谁是静态描述、谁是状态、谁在推进一步。” | 先去 `02_newton_arch` |
| “我能看懂一点例子，但 `wp.array`、`wp.launch` 这些词还没有稳定心智模型。” | 先去 `01_warp_basics` |
| “我不是卡在语法，而是卡在 `Model / State / Control / Solver` 怎么对应一条仿真执行链。” | 先去 `02_newton_arch` |
| “我看到 `articulation`、`M`、`J`、ABA、CRBA 时会慌，但暂时只是想知道它们在 Newton 架构里各负责什么。” | 先去 `02_newton_arch` |

最简单的路线是：

1. 如果你对 Warp 词汇还不稳，先读 `01`。
2. 如果你已经能接受 Warp 的最小执行模型，就直接去 `02`，把它落到 Newton 的 `Model / State / Control / Solver` 上。

这章的任务到这里就完成了。你不是来把前置一次学完的，你只是来把进入后两章最容易绊脚的几块地先铺平。
